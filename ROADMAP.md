## Constraints (non-negotiable, apply to every item)

- **stdlib-only Python 3.10+** — no runtime dependencies, ever.
- Must work with **small local models (3B–30B)** over **plain OpenAI-compatible chat APIs** (Ollama,
  vLLM, llama.cpp, MLX, LM Studio, TGI, SGLang, cloud). No engine-exclusive tricks without graceful
  degradation.
- Terminal UX, dependency-free TUI.
- The thesis governs design taste: **prefer mechanisms where the harness does work the model cannot** —
  deterministic scaffolding, verification, routing, context shaping. A prompt exhortation is a last resort.

6. Once **P3.2 (forge bench)** lands, every behavior-affecting item must ship with a bench ablation

## The five findings (what the audit revealed)

These cross-cutting truths motivate the phase structure:

1. **Verification is prose, not mechanism.** `VERIFY-BEFORE-DONE` is one system-prompt line; `say`
   returns unconditionally (`agent.py:275-280`). Real checking lives in an async daemon that must be
   running, fires only after the false claim, and its refutations rot in in-memory inboxes that idle
   sessions never drain (`repl.py:378-382`). → **P2**
2. **Context integrity has silent holes.** Post-compaction, read-before-edit passes on files whose
   content the model no longer has. Observations between 4k–12k chars are cut mid-line with no marker.
   OpenAI-compat streaming silently discards the usage frame, so compaction triggers off `chars//4`
   forever. → **P1, P4**
3. **The core thesis is unmeasured.** No benchmark, no ablation, no harness-lift number, no regression
   signal. Every lever change ships blind. → **P3**
4. **Constrained decoding is shallower than advertised.** One flat schema; only `thought`+`action` are
   grammar-required; per-action fields are unenforced; true grammar forcing exists only on Ollama. → **P5**
5. **Tool feedback is where harness intelligence should live, and mostly doesn't.** Syntax errors poison
   later steps undetected; test output arrives as 400 raw lines; failed edits demand "copy the EXACT
   text" instead of showing the nearest match. → **P1, P6**

## Phase 1 — Honest observations

Small, independent fixes that stop the harness silently lying to the model or to itself. Each is a self-contained PR with no dependency on anything else in this document. Do these first: every later phase builds on observations being truthful.
### P1.1 · Post-write syntax gate with auto-revert: the file is never left broken

**Status: done · Impact 5/5 · Effort S · Depends on: none** · note: check-before-write for write_file/edit_file via _syntax_error dispatch (py/json/toml + bash -n/node --check in temp files); invalid candidates are refused in-observation with a named error, real file untouched, ok=False — no revert path needed. toml guarded for py3.10 (no stdlib tomllib).

**What.** After every write_file/edit_file, the harness runs a per-extension stdlib check — compile() for .py, json.loads for .json, tomllib for .toml, `bash -n` for .sh, `node --check` for .js/.mjs when node exists — in the SAME observation as the write. For edit_file the check runs on the computed newtext BEFORE writing, so a failing edit is never written; for write_file the prior content is captured and restored on failure. The failure observation names the exact error and line ('that edit would make file.py invalid at line 42: unexpected indent — the file was NOT changed. Fix the snippet and retry.'), flips ok=False so it feeds fail accounting, and clean files get a one-word '(syntax OK)' so the model learns the invariant exists.

**Why.** A syntax error introduced by edit #1 poisons every subsequent step: later edits target a broken file and the eventual test run fails with a confusing traceback the model must map back to its own edit, many steps later, after the causal edit has scrolled toward compaction — the single most common death spiral for 3-9B models. Small models are dramatically better at fixing a named SyntaxError-with-line delivered in the same observation. Auto-revert/check-before-write makes 'the file is always syntactically valid' a harness invariant rather than a hope: the shortest possible verification loop, deterministic, zero model tokens — probably the highest value-per-line change available in the loop.

**Implementation sketch.** tools.py: _check_syntax(path, text) dispatching on extension — .py compile(text, path, 'exec') catching SyntaxError (has .lineno/.msg/.text for a quoted source line), .json json.loads, .toml tomllib.loads (try-import, 3.11+), .sh/.bash `bash -n` (2s timeout), .js/.mjs `node --check` iff _which('node') (helper at tools.py:16). edit_file branch (274-298): compute newtext, check, only write on pass, else return ok=False with the error (nothing written — no revert needed). write_file (266-273): capture prior content first (move the `before` capture out of agent.py:357-365 into tools so revert lives in one place), write, check, restore on failure. Keep read_files/ledger bookkeeping at agent.py:371-372 consistent for the checked-failed case. Emit checker name in ok observations ('edited x.py (exact, syntax OK)'). ~40 lines + tests mirroring the edit_file cases in tests/test_forge.py.

**Grounding (verified at `710f9d6`).** tools.py:266-273 (write_file), 274-298 (edit_file — newtext computed before write, perfect check point), tools.py:16 (_which); agent.py:357-365 (before-capture to relocate), 371-377 (post-execute bookkeeping), 380-390 (fail accounting it plugs into)

**Judge verification & corrections.** Gap fully confirmed: no syntax checking exists anywhere — write_file (tools.py:266-273) writes blind; edit_file (274-298) already computes newtext pre-write in the fuzzy path (293-297) and the exact path trivially can (text.replace(old,new,1) at 287-288 restructures to compute-then-check-then-write), making check-before-write clean with no revert needed for edits. _which exists (tools.py:16). Flipping ok=False integrates automatically: read_files bookkeeping (agent.py:371-372) is gated on ok, and fail accounting (380-390) picks it up with zero agent changes. CORRECTIONS to the proposal: (1) the before-capture at agent.py:357-365 cannot simply be 'moved' into tools — the diff event at :376 consumes it; either execute() returns before-text or tools keeps its own capture for revert (slight duplication, fine); (2) write_file revert of a previously-NONEXISTENT file must os.remove it, not restore empty string; (3) tomllib is 3.11+ and forge supports 3.10 — the proposed try-import is required; (4) bash -n / node --check need the candidate text in a temp file for the check-before-write path since the real file must not be touched on failure. compile() gives .lineno/.msg/.text for a quoted error line (IndentationError subclasses SyntaxError, covered). Impact 5 earned: deterministic, zero-model-token, same-observation named errors kill the poisoned-file death spiral that ends 3-9B multi-edit runs — the same mechanism class as Aider's lint-after-edit, which measurably improves weak-model edit success. Highest value-per-line in the batch.

**Merged variants:** Post-write syntax gate: compile/parse checks in the same observation as the write; Post-mutation syntax gate: the harness compiles every file the model writes, in the same observation

### P1.2 · One observation-shaping pipeline: end the 4k/12k split brain, keep tails, and make offloads reachable

**Status: done · Impact 4/5 · Effort S · Depends on: none** · note: added tools.shape(text,budget,note) head+marker+TAIL truncation with notes re-attached after the cut; _maybe_offload now returns (preview,note) and writes full text under <cwd>/.forge/output (auto .gitignore '*') so read_file/grep can follow the pointer; bg logs moved there too; agent.py derives one obs budget (~8% of effective_ctx, cap 12000) for both the transcript log and the fed-back message — both obs[:4000] slices gone.

**What.** Replace the two conflicting truncation layers with a single tools.shape(obs, notes, budget) used everywhere: budget derived from backend.effective_ctx() (e.g. min(12000 chars, ~8% of window)); truncation keeps head + TAIL with an explicit '[… N chars omitted from the middle …]' marker; harness notes (the offload pointer, '[showing lines X-Y of N]') are re-attached AFTER the cut so they can never be sliced off. Move the offload dir to <cwd>/.forge/output (auto-gitignored) so the model's own read_file/grep can actually follow the pointer it is given.

**Why.** The most damaging silent context bug in the pipeline: anything between 4k and 12k chars is cut mid-line with no marker (the model believes it saw everything — and read-before-edit is satisfied on 4k chars of a file); for >12k outputs the pointer telling the model where the full text lives is exactly what gets sliced off; and even when it survives, _resolve refuses the path as 'escapes the workspace'. Error messages and test summaries live at the END of output, so head-only truncation is the worst possible policy for a coding agent. Cheap fix that pays on every large observation for every model size.

**Implementation sketch.** tools.shape(text, notes, budget): keep budget*0.6 head + budget*0.3 tail + marker, then append notes verbatim. _maybe_offload (tools.py:86-102) returns (body, note) instead of a concatenated string; read_file's range note (tools.py:261-264) becomes a note too. agent.py:377 and 411 call tools.shape with budget = min(12000, effective_ctx-derived chars) — the one-size 4000 constant dies (transcript can keep its own cap). _OVERFLOW_DIR becomes overflow_dir(cwd) → <cwd>/.forge/output with a '.forge/' line in .git/info/exclude (or a .gitignore containing '*'); _resolve then confines it naturally. Background-process logs (tools.py:113-137) move too, so the model can tail them with read_file instead of bash-only.

**Grounding (verified at `710f9d6`).** agent.py:377,411 (obs[:4000] slices), tools.py:86-102 (_maybe_offload, 12000 budget, pointer appended last), tools.py:87 (_OVERFLOW_DIR), tools.py:190-205 (_resolve blocks ~/.forge/output), tools.py:113-137 (bg logs), tools.py:262-265 (range note)

**Judge verification & corrections.** Every cited defect confirmed, and one is worse than claimed: _maybe_offload caps at MAX_OUTPUT=12000 and appends the pointer AFTER the 12000-char head (tools.py:86-102), then agent.py:411 slices the whole observation to obs[:4000] — so for any offloaded output the pointer sits past char ~12000 and is ALWAYS cut; it is dead code from the model's perspective, meaning the entire offload mechanism currently delivers zero value through the agent loop. Same slicing kills read_file's '[showing lines X-Y of N]' note (tools.py:261-265) whenever the body exceeds 4000 chars. Confirmed _resolve confines to cwd (tools.py:190-205), so ~/.forge/output is unreachable by read_file/grep even when a pointer survives — only bash can follow it; bg logs (tools.py:113-137) share the problem. Confirmed 4000-6000-char observations are cut mid-line with no marker, and read-before-edit is satisfied by a partial 4000-char view (agent.py:371-372) — write_file overwrite is the real hazard there since edit_file at least requires an exact/fuzzy old-match. Head+tail retention is correct for a coding agent (errors and test summaries are tail-heavy). Effort correction: this is S, not M — refactor _maybe_offload to return (body, note), add tools.shape(), update two agent call sites (377 stays capped for the transcript if desired, 411 uses shape), move _OVERFLOW_DIR to <cwd>/.forge/output with a .gitignore, and update TOOL_HELP. Highest value-per-line of the batch.

### P1.3 · Failure autopsy: closest-match context on failed edits and error-class hints on failed bash

**Status: done · Impact 4/5 · Effort S · Depends on: none · note: edit_file now enumerates all match locations (exact AND >1-fuzzy branches), difflib sliding-window (0.5 floor, quick_ratio pre-filter, 4000-line cap) shows the CLOSEST region verbatim on 0-match, and failed bash appends the first of 11 (regex→hint) recovery lines; all paths still return ok=False to keep the escalation signal.**

**What.** When edit_file finds zero matches, the harness locates the most similar region of the file (difflib over sliding line windows) and includes it verbatim, with line numbers, in the failure observation; when `old` matches multiple times it lists each match location with one line of context. For failed bash, a small deterministic classifier maps common error signatures (ModuleNotFoundError, command not found, EADDRINUSE, No such file or directory) to one-line recovery hints appended to the observation.

**Why.** Failed edit_file is the #1 burn of small-model steps: the current 'copy the EXACT text' message forces a full re-read (another step, more context) before a retry that often fails again. Showing the actual nearby text converts a 3-step recovery loop into a 1-step fix, entirely with harness compute — the harness diagnoses deterministically so the model only has to act. Directly reduces the failure counts that trigger false escalations.

**Implementation sketch.** In the 0-match branch after _fuzzy_replace fails (tools.py:292-298), run difflib.SequenceMatcher over line windows of len(olines)±2 with a 0.5 ratio floor; on hit return 'edit failed: `old` not found. CLOSEST region in <path> (lines i-j):\n<region>\nRe-send edit_file with old copied EXACTLY from this.' In the n>1 branch (tools.py:290-291) enumerate match line numbers with one surrounding line. Bash: ~10 (regex, hint) pairs applied in _run's failure path (tools.py:173-176) or the failure-tagging block (agent.py:380-390) — e.g. ModuleNotFoundError → 'check for a venv (ls .venv/bin) and use its python, or pip install X'. Fixture-driven unit tests beside the existing _fuzzy_replace test.

**Grounding (verified at `710f9d6`).** tools.py:274-298 (edit_file failure paths), tools.py:68-83 (_fuzzy_replace), tools.py:150-176 (_run failure return), agent.py:380-390 (failure tagging)

**Judge verification & corrections.** Gaps confirmed exactly as claimed: the n>1 branch (tools.py:290-291) returns only a count with no locations; the zero-match branch (tools.py:292-298) falls through _fuzzy_replace to a generic 'Read the file and copy the EXACT text' with no nearby context, forcing a re-read step; bash failures (tools.py:173-176) and the failure tag (agent.py:380-390) carry no error-class hints. Directly reduces the total_fails counter (agent.py:382,393) that triggers escalation at STUCK_AT=7, so it genuinely cuts false escalations. Sharper detail found: _fuzzy_replace has a THIRD failure mode the proposal misses — '>1 fuzzy matches' (tools.py:79-80, distinct from the exact n>1 branch); it should get the same enumerate-locations treatment. Performance note: sliding-window SequenceMatcher over every position is O(n*m) on big files — pre-filter with real_quick_ratio or cap the scan window. Test scaffolding exists (test_fuzzy_replace at tests/test_forge.py:82). Pure stdlib difflib, no model calls, no engine dependencies. Best effort-to-impact ratio in the batch.

## Phase 2 — The verified loop

Convert VERIFY-BEFORE-DONE from prose into mechanism, end-to-end: a done-claim is harness-verified synchronously, independent verification is trustworthy, and a refutation autonomously wakes the session that made the claim. This closed loop is the product story. Note: P2.2 (Verifier v2) should land before or together with P2.3 (Wake-on-inbox) — wake-on-inbox makes REFUTED verdicts actionable, so false REFUTED must be fixed first.
### P2.1 · Synchronous done-gate: the harness runs the test before accepting `say`

**Status: done · Impact 5/5 · Effort M · Depends on: none** · note: `say` is now gated by a per-turn harness check — if the turn mutated files and nothing verified them, detect_test_cmd is run in-process and the exit code either lets the say through (logging a 'verified' record the daemon honors to skip its rsync verify) or bounces it once with the real failing tail; no-suite/exit-127 accept gracefully. Mutation tracking counts file-touching BASH too (fleet.bash_mutates: redirects, sed/perl -i, rm/mv/cp/git checkout…), so `echo x > app.py` / `sed -i` can't slip past the gate — and harness_verified invalidates a stale 'verified' record on any such bash edit so the daemon still independently re-verifies. `_is_test_cmd` is anchored at a command HEAD (not a substring), so `pytest --version`, `which pytest`, `pip install pytest`, and `git commit -m "make test green"` no longer falsely mark the turn verified. Deviations: plan-incomplete bounce and the ask-once-for-a-demo-command path are not implemented (no detectable suite silently accepts); the gate is on-by-default and runs the suite inline. Full suite green (144 tests).

**What.** Gate the turn-ending `say` mechanically: if the turn mutated files and nothing has verified them, the HARNESS runs fleet.detect_test_cmd(cwd) itself — zero model tokens — and either lets the say through (appending the pass evidence and writing a 'verified' transcript record) or bounces it back exactly once with the real failing output as an observation. A second say in the same turn always passes (no livelock) but is tagged. Also bounce once if the plan still has [ ]/[~] items (naming them), or — when no test command is detectable — ask once for a demonstrating command. A 'verified' record lets the daemon's verify_pass skip its expensive rsync verify for already-harness-verified claims.

**Why.** VERIFY-BEFORE-DONE is currently one line of prose plus an async daemon that must be running, fires only after the lie is told, and whose refutations rot in idle sessions' in-memory inboxes. Small models over-claim constantly; this converts the claim into a harness-verified fact synchronously, per turn, while the model still has full context to fix the failure — the exit code is the verdict, no model judgment involved. The single change most likely to move end-to-end task success for 3-9B models, because unverified false completion is their dominant end-state failure.

**Implementation sketch.** Per-turn state reset near agent.py:241-244: self._mutated=set(), self._verified=False, self._bounced=False. ## Appendix B — Blind spots the completeness critic caught

The first ideation round (7 lenses, 64 proposals) entirely missed these angles; the gap-fill round
produced the P7/P8 items above. Recorded so future planning starts from them:

- Defense-in-depth for an unconfined autonomous agent: today bash has full machine access while file tools are carefully workspace-confined, the agent's system prompt declares fleet messages 'trusted; read it and act' even though any local process that reads ~/.claude/fleet/token can POST straight into every inbox (a local prompt-injection-to-code-execution path the map itself flags), and nothing redacts secrets — observations, JSONL transcripts, offload files, and LEARN facts persist raw command output (env dumps, .env reads, tokens in error messages) to ~/.forge forever and re-inject it into future sessions' prompts. Ideate a security layer proposal: OS-level bash confinement profiles (sandbox-exec on macOS, bwrap/unshare on Linux, degrading to the classifier-only mode elsewhere) that jail bash to the workspace plus an explicit allowlist; an untrusted-content boundary that wraps inbound fleet messages and tool output in 'data, not instructions' framing with mutating-action approval required when acting on them; and a deterministic secret scrubber (regex for key/token/bearer patterns + values of known env vars) applied before anything is written to transcripts, offload files, or the LEARN store. Note the existing 'read-only bash classifier' proposal covers none of this — it only decides what skips approval, not what a command can touch once approved.
- Infrastructure fault tolerance — surviving the backend, not just the task: the map shows a ForgeError at stream-start propagates uncaught out of Agent.send() (one Ollama restart kills the whole turn with a traceback path), a mid-stream disconnect returns partial text that gets misattributed as a model 'malformed JSON' strike, Esc abandons the HTTP response while the server keeps generating (burning GPU and queueing the next request behind the orphan), second-Esc leaks a live worker thread that keeps mutating agent.messages while a new send starts, and escalation never fires on infra failures because only tool failures count. Ideate a resilience layer: classify exceptions into infra-vs-model at the _generate boundary; retry infra failures with bounded backoff (and an honest '[backend unreachable — retrying 2/3]' event) instead of charging the model a strike; wait-and-reconnect when Ollama restarts mid-session (re-warm, re-probe window, invalidate the cached-on-failure 8192 context); actively cancel orphaned generations on stop (close the connection, drain-then-close); and a generation-counter guard so a superseded worker thread's events and message-appends are discarded rather than interleaved. None of the checkpoint/resume/daemon-hardening proposals touch the live backend-failure path — they all assume the process and engine stay up.
- First-run and day-one experience — the road from 'pip install' to a successful first task: no proposal touches setup.py or onboarding at all, yet the map shows setup writes config without ever pinging the server, pulls models whose failure still lands them in the ladder, hardcodes a 2026 model-name snapshot, tiers purely on RAM with zero GPU/VRAM detection, and a session can start fully blind because _workspace_ctx swallows every exception silently. Ideate the onboarding proposal: a `forge doctor` subcommand (engine reachable? models actually served — via /api/tags or /v1/models? schema/constrained-decoding dialect supported? window probe sane? fleet daemon up? bridge token present?) with fix-it one-liners per failure; setup that validates before writing (ping URL, confirm typed ladder names against served models, run the 1-token schema probe, refuse to write a ladder entry whose pull failed); in-session graceful degradation when the engine is down at launch ('ollama isn't running — start it? [y/n]' instead of a ForgeError); VRAM/Metal-aware sizing; and a 60-second guided first task that demonstrates plan mode, an approval prompt, and a verified done — the make-or-break moment for a tool whose pitch is 'works on your laptop'.
- Tool-surface extensibility and ecosystem interop — the gap a Claude Code user hits in the first five minutes: forge's tool surface is 9 hardcoded actions with no web access of any kind (no fetch, no search — only raw curl-via-bash with no HTML-to-text, no offload-aware shaping) and no way to add a tool without editing tools.py; the 'open lever API' proposal extends harness hooks but explicitly not the action set. Ideate the extensibility proposal: a minimal stdlib-only MCP client (stdio JSON-RPC subprocess per server, configured in ~/.forge/config.json) that merges each MCP tool into the action grammar as a first-class schema variant — so constrained decoding, the mode gate (MCP tools default to mutating/approval-required), loop signatures, and observation shaping all apply to external tools for free; plus one built-in `web_fetch` action (urllib + a small HTML-to-markdown reducer, output routed through the offload pipeline) since small local models can't be trusted to hand-roll curl+parse.
